## Key Findings

1. **Gateway is production-ready** at 50 RPS sustained (5 min, 0% errors, p99 297ms)
2. **Comfortable ceiling at 300 RPS** on shared-cpu-1x (p99 ~1s, <0.1% errors)
3. **Rate limiter is the bottleneck**, not CPU — the `tokio::sync::Mutex<HashMap>` starts causing 429s around 350+ RPS from a single client
4. **Latency is dominated by xAI response time** (~200-400ms) — gateway overhead is minimal (~10-60ms based on p50 variance across RPS levels)
5. **Provider prefix bug found and fixed** — OpenAI, xAI, DeepSeek were sending prefixed model names (e.g., `openai/gpt-4o-mini`) to upstream APIs; only Google and Anthropic stripped their prefixes

## Scaling Recommendation

| Expected Load | Recommended Tier | Notes |
|---------------|-----------------|-------|
| < 50 RPS | shared-cpu-1x / 512MB (current) | Production-ready as-is |
| 50-200 RPS | shared-cpu-1x / 512MB | Comfortable with headroom |
| 200-300 RPS | Consider performance-2x | p99 starts rising above 1s |
| 300+ RPS | Redesign rate limiter first | Mutex contention is the bottleneck, not CPU |
